Why DNS TXT Most resilient verification anchor

Of the five verification methods Google offers (HTML file, HTML meta tag, GA, GTM, DNS TXT), DNS TXT is the only one decoupled from the web application. It survives:

  • Django template rewrites that strip <meta> tags
  • Whitenoise CompressedManifestStaticFilesStorage hash-renaming the verification HTML file
  • Full server re-deploys or even a hosting move

The TXT record lives in cPanel's Zone Editor, scoped to the root bejundas.co.tz zone. Bing then re-uses the same trust chain via OAuth import, so no second DNS record is needed.

Sitemap Architecture

§2

Two django.contrib.sitemaps.Sitemap subclasses are mounted under a single /sitemap.xml endpoint via the sitemaps dict in config/urls.py. Output validated against Google Search Console schema — "Discovered pages: 11" matches the static sitemap's STATIC_PAGES length exactly, confirming no News items are currently published.

# apps/hub/sitemaps.py
from django.contrib.sitemaps import Sitemap
from django.urls import reverse
from apps.hub.models import News

STATIC_PAGES = [
    ("hub:home",             1.0, "weekly"),
    ("hub:about",            0.8, "monthly"),
    ("hub:services",         0.9, "monthly"),
    ("hub:news",             0.7, "weekly"),
    ("hub:team",             0.6, "monthly"),
    ("hub:contact",          0.8, "monthly"),
    ("leads:financial",      0.7, "monthly"),
    ("leads:construction",   0.7, "monthly"),
    ("leads:energies",       0.7, "monthly"),
    ("leads:farming",        0.7, "monthly"),
    ("leads:investments",    0.7, "monthly"),
]

class NewsSitemap(Sitemap):
    changefreq = "weekly"
    priority   = 0.6

    def items(self):
        return News.objects.filter(published=True).order_by("-published_at")

Static pages get hand-tuned priorities — home at 1.0, Services at 0.9 (primary commercial intent), About + Contact at 0.8, vertical landing pages at 0.7, Team at 0.6. The News sitemap is dynamic and filters on published=True so unpublished drafts are never exposed to crawlers.

robots.txt

§3

Served by a function-based view in config/urls.py — not a static file. This lets the host be read dynamically from the request, so the sitemap URL line stays correct across local dev (127.0.0.1:8000) and production (bejundas.co.tz) without an env-var swap.

# config/urls.py
def robots_txt(request):
    lines = [
        "User-agent: *",
        "Allow: /",
        "Disallow: /admin/",
        "Disallow: /deploy/",
        f"Sitemap: https://{request.get_host()}/sitemap.xml",
    ]
    return HttpResponse("\n".join(lines), content_type="text/plain")

Disallow rationale:

  • /admin/ — Django admin login surface. No SEO value, just a probing target for bots if indexed.
  • /deploy/ — webhook endpoint (apps.core.urls_webhook). Crawling it triggers HMAC mismatch responses; pointless to surface.

Decisions & Trade-offs

§8
DNS TXT over HTML file verification Trade-off: requires DNS edit access. Win: survives Whitenoise's manifest hashing which would rename any verification HTML file and break trust.
Bing import from GSC over separate verification Trade-off: couples Bing's trust chain to the Google account. Win: one less DNS record to manage, sitemap re-shared automatically.
robots.txt as a view, not a static file Trade-off: tiny per-request cost (negligible). Win: the Sitemap: line is host-aware. Same code works in dev, staging, and prod without environment-specific files.
Vertical Coming Soon pages included in sitemap Trade-off: indexes pages that don't have unique content yet (they share one template). Win: when a vertical ships, Google has already crawled the URL — replacement content is picked up in the next pass rather than waiting for fresh discovery.
